## 1. 要件定義

### 1.1 ユーザーストーリー

- ユーザーは日々「気になっていること・不安なこと」を1行で登録する
- 登録後、時間経過とともに「まだ何も起きていません」という通知が段階的に届く
- 24時間経過すると「穏やかな一日でした」という締めのメッセージが届く
- 過去の「穏やかだった日」がカレンダー形式で蓄積され、振り返れる

### 1.2 機能要件(MVP範囲)

| ID | 機能 | 優先度 |
|---|---|---|
| F-01 | 不安リストの登録(テキスト入力のみ) | 必須 |
| F-02 | 不安リストの一覧表示・削除 | 必須 |
| F-03 | 登録から6h/12h/24h後のローカル通知 | 必須 |
| F-04 | 通知文言のランダム生成(同一状況でも表現が変わる) | 必須 |
| F-05 | 24h経過時点で自動的に「穏やかな日ログ」を生成 | 必須 |
| F-06 | 穏やかな日ログのカレンダー表示 | 必須 |
| F-07 | 日ごとの詳細確認(どの不安が何件穏やかに終わったか) | 中 |
| F-08 | ダークテーマ + セリフ体フォントのUI世界観 | 中 |
| F-09 | オンボーディング(初回起動時の説明画面) | 低 |

### 1.3 非機能要件

| ID | 内容 |
|---|---|
| NF-01 | 通信を一切行わない(プライバシー完結・審査/課金対応不要) |
| NF-02 | ローカル通知は端末あたり最大64件の制約を考慮した設計にする |
| NF-03 | SwiftDataでのローカル永続化、マイグレーション対応を考慮 |
| NF-04 | 通知権限が拒否された場合のフォールバックUI(アプリ内表示のみで完結) |
| NF-05 | アクセシビリティ(Dynamic Type対応、VoiceOver最低限対応) |

### 1.4 スコープ外(MVPでは作らない)

- SNS連携・実際の通知監視(技術的に不可能なため対象外)
- 複数端末同期(iCloud同期)
- プッシュ通知(サーバー不要のためローカル通知のみ)
- 課金機能

## 3. データ設計 (SwiftData)

```swift
import Foundation
import SwiftData

// 不安リストの1項目を表すモデル
@Model
final class WorryItem {
    @Attribute(.unique) var id: UUID       // 一意識別子
    var text: String                        // ユーザーが入力した不安の内容
    var createdAt: Date                     // 登録日時(通知スケジュールの基準)
    var isCompleted: Bool                   // 24h経過して「穏やか」認定されたか

    init(text: String) {
        self.id = UUID()
        self.text = text
        self.createdAt = Date()
        self.isCompleted = false
    }
}

// 1日ごとの「穏やかだった日」ログ
@Model
final class PeacefulLog {
    @Attribute(.unique) var date: Date      // ログ対象の日付(日単位で丸める)
    var completedWorryTexts: [String]       // その日に完了した不安の内容(スナップショット保存)

    init(date: Date, completedWorryTexts: [String]) {
        self.date = date
        self.completedWorryTexts = completedWorryTexts
    }
}
```

**設計上の注意点(NF-03対応):**
- `WorryItem`は完了後も削除せず`isCompleted`で管理 → 後から履歴を追える
- `PeacefulLog`は日付ごとに1件、配列でスナップショットを持つことで正規化コストを避ける(個人開発規模ならこれで十分)
- 将来的にフィールド追加する場合はSwiftDataの`VersionedSchema`でマイグレーション対応を検討

## 4. 技術設計

### 4.1 アーキテクチャ

**MVVM + Repositoryパターン**を採用(ポートフォリオとしての見せ方も考慮)

```
SilentDay/
├── App/
│   └── SilentDayApp.swift          // エントリーポイント、ModelContainer初期化
├── Models/
│   ├── WorryItem.swift
│   └── PeacefulLog.swift
├── Repositories/
│   ├── WorryRepository.swift       // WorryItemのCRUD処理を抽象化
│   └── LogRepository.swift         // PeacefulLogのCRUD処理を抽象化
├── Services/
│   ├── NotificationService.swift   // ローカル通知のスケジュール管理
│   └── MessageGenerator.swift      // 通知文言のランダム生成ロジック
├── ViewModels/
│   ├── HomeViewModel.swift
│   ├── AddWorryViewModel.swift
│   └── CalendarViewModel.swift
├── Views/
│   ├── HomeView.swift              // S-01
│   ├── AddWorryView.swift          // S-02
│   ├── NotificationDetailView.swift // S-03
│   ├── CalendarView.swift          // S-04
│   ├── DayDetailView.swift         // S-05
│   └── OnboardingView.swift        // S-06
├── DesignSystem/
│   ├── Colors.swift                 // ダークトーンのカラーパレット定義
│   ├── Typography.swift             // セリフ体フォント設定
│   └── Animations.swift             // フェードイン等の共通アニメーション
└── Resources/
    └── Info.plist                   // 通知権限の利用目的文言など
```

### 4.2 通知スケジュール設計(NF-02対応)

- 1つの`WorryItem`につき通知は最大3件(6h/12h/24h後)のみスケジュール
- 登録数上限を**同時最大10件**にUI側で制限 → 最大30件の保留通知で64件制約に余裕を持たせる
- `WorryItem`削除時は対応する通知もキャンセルする(`UNUserNotificationCenter.removePendingNotificationRequests`)

### 4.3 通知文言生成ロジック(F-04対応)

- 経過時間帯(0-6h / 6-24h / 24h以降)ごとにテンプレート配列を用意
- `randomElement()`でランダム選択し、同じ状況でも表現の揺らぎを出す
- 将来的にテンプレート数を増やす際は`MessageGenerator`に配列を追加するだけで拡張可能な設計にする

## 5. UI/UX トーン&マナー

- カラー:ダークネイビー〜チャコールを基調(明るい色は避ける)
- フォント:游明朝 / Georgia等のセリフ体を優先使用
- アニメーション:即時表示を避け、0.5〜1秒のフェードインを基本にする
- 通知文言:断定的すぎず、静かに寄り添うトーン(例:「まだ静かな時間が続いています」)

## 7. 注意点まとめ

- Info.plistに`NSUserNotificationsUsageDescription`相当の説明文言を必ず用意(実機テスト時に権限リクエストが必要)
- 通知許可が拒否されても機能不全にならないよう、アプリ内表示だけで完結するフォールバックを用意
- 通知文言のトーンが暗くなりすぎないよう、テンプレート作成時に見直しを入れる
- 無料Apple IDでの実機ビルドは証明書7日期限があるため、頻繁な再ビルドが必要

## 11. v1.2 追加要件:振り返り機能(統計・グラフ表示)

v1.1完了後の追加開発。既存データ(WorryItem, PeacefulLog)を活用し、新規の書き込み処理は発生しない、読み取り専用の追加機能。

### 11.1 背景・目的

- v1.0〜v1.1では「登録する」「通知を受け取る」「カレンダーで振り返る」までは実装済みだが、蓄積されたデータを俯瞰する手段がない
- 「自分がどれくらい静かな時間を過ごせているか」を可視化することで、アプリのコンセプト(安心の記録)をより実感できるようにする

### 11.2 要件

| ID | 内容 |
|---|---|
| V1.2-01 | カレンダー画面(S-04)に「統計」タブまたはボタンを追加し、新画面(S-07 統計画面)へ遷移できるようにする |
| V1.2-02 | 直近30日間の「穏やかだった日」の推移を棒グラフまたは折れ線グラフで表示する(Swift Charts使用) |
| V1.2-03 | 累計の見守り件数(WorryItem総数)、累計の穏やかな日数(PeacefulLog総数)をサマリー表示する |
| V1.2-04 | 平均して何時間で「穏やか」と判定されたか(常に24時間固定のため、この指標は将来の可変対応を見据えた表示に留める、または今回は割愛可) |
| V1.2-05 | データが少ない(統計を出すには不十分な)場合は、詩的な空状態メッセージを表示する(既存のemptyStateAreaのトーンを踏襲) |

### 11.3 画面設計:S-07 統計画面(新規)

| 項目 | 内容 |
|---|---|
| 役割 | 蓄積したデータを俯瞰し、達成感・安心感を提供する |
| 主要コンポーネント | サマリーカード(累計見守り件数・累計穏やかな日数)、直近30日の棒グラフ、空状態メッセージ |
| 遷移元 | カレンダー画面(S-04)のツールバーまたはタブから遷移 |
| デザイン | 既存のダークトーン・明朝体見出しを踏襲。グラフの配色はAccentMain(#8B7FD1)を基調とする |

### 11.4 技術方針

- グラフ描画にはSwift Charts(iOS 16以降、標準フレームワーク)を使用する。追加ライブラリ不要
- データ集計はLogRepository/WorryRepositoryの既存メソッドを流用し、新規の書き込みロジックは発生させない
- CLAUDE.mdの方針どおり、絞り込みには`#Predicate`を使わず、全件取得後にSwift側でfilter/集計する
- ViewModelは`StatisticsViewModel.swift`としてViewModelsフォルダに新規作成し、Viewは`StatisticsView.swift`としてViewsフォルダに新規作成する(既存のMVVM構成を踏襲)
